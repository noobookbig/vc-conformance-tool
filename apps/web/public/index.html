About

What this is, what it isn't, and how to read the report.

The VC Conformance Test Webapp is a self-contained tool for QA engineers who need to verify an OpenID for Verifiable Credential Issuance 1.0 Final issuer or an OpenID for Verifiable Presentations 1.0 Final verifier — and to verify wallets against real targets.

It is not a wallet product, a registry, or a trust-list authority. It does not store PII. It does not commit to a real Verifiable Data Registry. It is a test harness.

Cross-modes

  • Issuer ↔ Wallet — drive a target Issuer with our wallet simulator. Use this to verify an Issuer you control. Tests the OID4VCI issuance pipeline (offer → authorization → token → credential).
  • Verifier ↔ Wallet — drive a target Verifier with our wallet simulator. Use this to verify a Verifier you control. Tests the OID4VP presentation request and response.
  • Wallet ↔ Issuer — drive our wallet simulator with a target Issuer. Use this to test your wallet against a real Issuer.
  • Wallet ↔ Verifier — drive our wallet simulator with a target Verifier. Use this to test your wallet against a real Verifier.

Coverage

The webapp implements the happy-path + canonical negative-path of every operation the spec defines for each role. The current concrete suite is roughly 28 executable test cases covering credential offer, authorization (PKCE + authorization_details), token exchange, credential request (KB-JWT ES256 + EdDSA), deferred credential, notification, and OID4VP DCQL presentation. The number of concrete cases per mode is configurable.

How to read a report

Each row in the report is one test. ✅ means the spec-shaped request was built correctly by our simulator and the target accepted (or rejected, for negative tests). ⏭️ means a prerequisite for that test was not met (e.g. you ran Verifier mode but didn't supply a target verifier). ❌ means the simulator could not build the request per the spec, or the target returned an unexpected error.
